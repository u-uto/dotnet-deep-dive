# 01_program-structure  ：プログラム構造

## 概要

このファイルでは、C# プログラムの基本構造を整理します。  
ここで扱うのは、C# のコードを「どう書くか」ではなく、まず **どのような骨格で組み立てるか** です。

C# のプログラムは、単に処理を上から下に並べるだけではありません。  
実務では、次のような観点で構造を理解する必要があります。

- どこから実行が始まるのか
- `namespace` をどう切るのか
- `class` をどう配置するのか
- `Main` とトップレベルステートメントをどう見るのか
- ファイルをどう分けるのか

このファイルを理解すると、C# のサンプルや既存コードを見たときに、  
「どこが入口で、どこが処理本体か」を追いやすくなります。

---

## 定義

### プログラム構造とは何か

ここでいう **プログラム構造** とは、C# ソースコード全体をどのような入れ物で構成するかということです。  
具体的には、次の要素を含みます。

- アプリケーションの実行開始点
- 名前空間の切り方
- クラスとメソッドの配置
- ファイルの分割方法
- コードの責務分担

### エントリーポイントとは何か

**エントリーポイント** は、アプリケーションの実行が開始される場所です。  
従来の C# では `Main` メソッドがそれにあたります。  
一方で、近年の C# ではトップレベルステートメントを使うことで、明示的な `Main` を書かない形もあります。

### 名前空間とは何か

**namespace** は、型やメンバーの名前衝突を避けるための論理的な区画です。  
大規模なコードほど、名前空間の整理は重要になります。

### クラスとは何か

**class** は、データや処理をまとめる基本単位です。  
C# では、多くの場合、機能の中心はクラスとして表現されます。

---

## 特徴

### 1. C# には複数の書き方がある
同じアプリケーションでも、従来型の `Program` クラス + `Main` で書く方法と、トップレベルステートメントで書く方法があります。  
どちらも正しいですが、用途やチームの方針で使い分けます。

### 2. 構造は読みやすさに直結する
コードの構造が整っていると、実行点、依存関係、責務の切り分けが分かりやすくなります。  
逆に、構造が曖昧だと、短いコードでも追いづらくなります。

### 3. 実務では「小さい単位で分ける」ことが重要
1ファイルにすべてを詰め込むのではなく、クラス、責務、名前空間の単位で分けるのが基本です。  
この考え方は、ASP.NET Core やライブラリ開発でもそのまま活きます。

### 4. 学習初期に理解すると後が楽になる
構造を早めに押さえると、`types/`、`generics/`、`oop/`、`async-await/` へ進んだときに、コードの全体像を見失いにくくなります。

## 代表例

### 1. トップレベルステートメント

```csharp
using System;

Console.WriteLine("Hello, world!");
```

これは最も短い C# プログラムの形です。  
学習や小さな確認用には非常に便利です。

---

### 2. `Program` クラス + `Main`

```csharp
using System;

namespace DemoApp
{
    internal class Program
    {
        static void Main(string[] args)
        {
            Console.WriteLine("Hello, world!");
        }
    }
}
```

従来からある標準的な形です。  
構造が明示的なので、学習用としても理解しやすいです。

---

### 3. ファイルスコープ namespace を使う形

```csharp
using System;

namespace DemoApp;

internal class Program
{
    static void Main(string[] args)
    {
        Console.WriteLine("Hello, world!");
    }
}
```

中括弧のネストを減らせるため、現代の C# ではよく使われます。

---

### 4. 複数クラスを分ける形

```csharp
namespace DemoApp.Services;

internal class GreetingService
{
    public string GetMessage()
    {
        return "Hello, world!";
    }
}
```

```csharp
using DemoApp.Services;

namespace DemoApp;

internal class Program
{
    static void Main(string[] args)
    {
        var service = new GreetingService();
        Console.WriteLine(service.GetMessage());
    }
}
```

この形では、責務をクラスごとに分ける考え方が見えます。

## 使いどころ

プログラム構造の理解は、次の場面で特に役立ちます。

- C# の最初の学習
- サンプルコードの読み取り
- コンソールアプリの作成
- ASP.NET Core の入口理解
- ライブラリや小規模ツールの設計
- チームでのファイル分割方針の整理

特に実務では、「どこから実行が始まるのか」「どのクラスが何を担当しているのか」をすぐ追えることが重要です。

## 注意点

### 1. トップレベルステートメントを知っていても `Main` を理解しておく
今の C# ではトップレベルステートメントが便利ですが、古いコードや一部のサンプルでは `Main` が使われています。  
両方を読めるようにしておくと、実務で困りません。

### 2. `namespace` とフォルダ構成は一致させることが多い
必須ではありませんが、実務ではフォルダ構造と namespace をある程度揃えると保守しやすくなります。

### 3. 学習時は構造を単純に保つ
最初から複雑な DI、レイヤード構成、複数プロジェクト構成にしすぎると、構文理解がぼやけます。  
この章では、まず最小構成を理解することを優先します。

### 4. `using` の役割を雑に扱わない
`using` は単なる省略記法ではなく、参照の整理に関わる重要な要素です。  
後続の `using-directives.md` 章でもさらに扱います。

## 実装例

### 例1: 最小のアプリ

```csharp
Console.WriteLine("Hello, world!");
```

これは「とりあえず動かす」ための最小例です。  
短さのために学習しやすいですが、構造を理解するには少し情報が少ないです。

---

### 例2: 明示的な構造を持つコンソールアプリ

```csharp
using System;

namespace DemoApp
{
    internal class Program
    {
        static void Main(string[] args)
        {
            Console.WriteLine("Start");

            var message = "Hello, world!";
            Console.WriteLine(message);

            Console.WriteLine("End");
        }
    }
}
```

この例では、次の点が見やすくなります。

- `namespace` の位置
- `class` の位置
- `Main` が実行開始点であること
- 処理の流れ

---

### 例3: 責務を分けた形

```csharp
namespace DemoApp.Services;

internal class GreetingService
{
    public string CreateGreeting(string name)
    {
        return $"Hello, {name}!";
    }
}
```

```csharp
using DemoApp.Services;

namespace DemoApp;

internal class Program
{
    static void Main(string[] args)
    {
        var service = new GreetingService();
        Console.WriteLine(service.CreateGreeting("Taro"));
    }
}
```

この形は、後で `oop/` や `architecture/` に進んだときの入口にもなります。

## 設計指針

### 1. 入り口はシンプルにする
この章では、複雑な設計よりも「全体の見通し」を優先します。  
学習者が最初に見るものとして、ノイズを減らすことが大切です。

### 2. 入口と責務を分ける
`Program` は入口、別クラスは処理担当、という分け方を基本にします。  
この考え方は大きなアプリケーションでも重要です。

### 3. 1ファイル1役割を意識する
構造が見えやすくなるだけでなく、後から差分を追いやすくなります。

### 4. 古い形式と新しい形式の両方を見せる
現行の C# を学ぶときでも、既存コードの読み解きのために従来形式は必要です。  
そのため、トップレベルステートメントだけで終わらせず、`Main` 形式も必ず扱います。

## 判断基準

この章でどこまで深く書くかは、次の基準で決めます。

### 1. 実務でよく見るか
`Main`、`namespace`、`using` は頻出なので、丁寧に扱います。

### 2. 後続章の前提になるか
プログラム構造は、型、メソッド、OOP、ASP.NET Core の前提になるので重要です。

### 3. コード例が短く見せられるか
短くても意味がある例を優先します。  
長大な例はこの章には向きません。

### 4. 学習者が混乱しやすいか
トップレベルステートメントと `Main` の違いは混乱しやすいので、比較して示す価値があります。

---

## よくある誤解

### 1. `Main` がないと動かない
現在の C# では、トップレベルステートメントを使えば `Main` を明示しない形でも動きます。

### 2. `namespace` は必ず大きな意味を持つ
`namespace` は主に名前の整理のための仕組みです。  
設計上の責務分割とは関係しますが、同じものではありません。

### 3. フォルダ構成と namespace は完全一致でなければならない
完全一致は必須ではありません。  
ただし、実務では揃えると分かりやすくなります。

### 4. 小さなサンプルでは構造を意識しなくてよい
小さいコードでも、構造の癖はそのまま身につきます。  
最初から丁寧に書くことが重要です。
