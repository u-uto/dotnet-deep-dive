# 識別子
02_identifiers

## 概要

このファイルでは、C# における **識別子** を整理します。  
識別子とは、変数名、メソッド名、クラス名、名前空間名など、コードの中で対象を区別するために付ける名前のことです。

C# を読む力は、文法を知るだけでは足りません。  
**名前をどう読むか** を理解すると、コードの意味が一気に追いやすくなります。  
そのため、このファイルでは「どう書くか」だけでなく、「どう見ればよいか」も重視します。

このファイルで扱う内容は次のとおりです。

- 識別子の意味
- 命名ルール
- 命名規約
- 大文字・小文字の使い分け
- 予約語との衝突
- 実務での命名の考え方
- よくある命名ミス

---

## 定義

### 識別子とは何か

**識別子** は、プログラム内の要素を区別するための名前です。  
たとえば、次のようなものが識別子にあたります。

- 変数名
- 定数名
- メソッド名
- クラス名
- プロパティ名
- 引数名
- 名前空間名

### 識別子の役割

識別子の役割は、単に「名前をつける」ことではありません。  
コードを読む人が、その要素の意味や用途を推測できるようにすることです。

たとえば `x` という名前より、`totalPrice` のほうが意味を読み取りやすいです。  
識別子は、コードの可読性に直接影響します。

---

## 特徴

### 1. 識別子は意味を伝える
よい識別子は、コードの意図を自然に伝えます。  
名前だけである程度の役割が分かると、実装を追いやすくなります。

### 2. 名前は短すぎても長すぎてもよくない
短すぎる名前は意味が曖昧になります。  
長すぎる名前は読みづらくなります。  
適切な長さで、役割が分かることが大切です。

### 3. C# には命名の慣習がある
C# では、用途ごとに大まかな命名規約があります。  
これを守ると、コード全体が自然にそろいます。

### 4. 識別子はチーム開発で特に重要
個人開発では自分だけが理解できればよい場面もありますが、実務では他人が読む前提です。  
そのため、命名は設計の一部と考えるべきです。

## 使いどころ

識別子の理解は、次の場面で特に役立ちます。

- コードを読んで意味を追うとき
- 新しい変数やメソッドを作るとき
- 既存コードの責務を推測するとき
- リファクタリングするとき
- チームで命名規約をそろえるとき

特に実務では、**名前を見て何をするコードか分かるか** が非常に重要です。

---

## 注意点

### 1. 意味のない名前を避ける
`data`、`temp`、`value` のような名前は、場面によっては便利ですが、濫用すると意味が曖昧になります。  
短い名前を使うなら、その場で十分に意味が通るかを確認します。

### 2. 型が分かればよいとは限らない
`string name` のように型が分かっても、何の名前なのかまでは分かりません。  
識別子は型情報の補助ではなく、意味情報の中心です。

### 3. 略しすぎない
略語は便利ですが、読み手に伝わらない略し方は避けます。  
社内で通じる略語でも、あとで分からなくなることがあります。

### 4. 一貫性を崩さない
同じ意味のものに別の命名をすると、コードが散らかって見えます。  
たとえば `userId` と `customerId` が混在していると、役割の違いが曖昧になります。

## 実装例

### 例1: わかりやすい識別子

```csharp
int userAge = 25;
string customerName = "Sato";
decimal orderAmount = 9800m;
```

このように、名前を見ただけで用途が推測しやすいと、コードは読みやすくなります。

---

### 例2: わかりにくい識別子

```csharp
int a = 25;
string b = "Sato";
decimal c = 9800m;
```

短く書けますが、何を表しているのかが分かりにくくなります。  
短いコードでも、後から読むときに負担になります。

---

### 例3: 関数の意図が分かる命名

```csharp
bool IsAdult(int age)
{
    return age >= 18;
}
```

`IsAdult` という名前だけで、戻り値が真偽値であることや、何を判定するかが見えてきます。

---

### 例4: クラス名と責務

```csharp
internal class InvoiceCalculator
{
    public decimal Calculate(decimal subtotal, decimal tax)
    {
        return subtotal + tax;
    }
}
```

クラス名から、請求書計算に関係する責務を持つことが分かります。

## 設計指針

### 1. 名前は役割を表す
識別子は「何であるか」を表します。  
「見た人が役割を推測できる名前」を優先します。

### 2. 一貫した命名規約を守る
C# では、用途ごとに命名の傾向をそろえます。  
個人の好みよりも、チームや言語の慣習を優先したほうが読みやすくなります。

### 3. 省略を最小限にする
一時的には短くても、後から意味が消えない名前にします。

### 4. 名前の責務を分ける
`data` のような汎用名で何でも表すのではなく、意味ごとに切り分けます。  
たとえば `inputText`、`parsedValue`、`formattedMessage` のように分けると、処理の流れが見えます。

---

## 判断基準

識別子を付けるときは、次の基準で判断します。

### 1. 名前だけで役割が分かるか
最低限、何を表しているかが伝わる必要があります。

### 2. 長すぎないか
説明的すぎて読みにくくならないようにします。

### 3. 略語が通じるか
略語は、誰が見ても意味が分かる場合だけ使います。

### 4. 他の識別子と衝突しないか
近い名前が多すぎると、読み手が混乱します。

### 5. C# の慣習に合っているか
言語に合った命名を使うことで、自然な見た目になります。

---

## よくある誤解

### 1. 名前は自分が分かればよい
実務では自分以外も読むため、それでは不十分です。

### 2. 短い名前のほうが上級者っぽい
短い名前が優れているとは限りません。  
意味が伝わることのほうが重要です。

### 3. 型が分かれば十分
型だけでは意図が分からないことが多いです。  
識別子の役割はそれを補うことです。

### 4. 名前は後で直せばよい
もちろん修正はできますが、最初からよい名前を付けるほうが圧倒的に効率的です。

## まとめ

識別子は、C# の読みやすさを支える重要な要素です。  
文法としては単純に見えても、実務ではかなり大きな差を生みます。

このファイルで押さえるべき点は次のとおりです。

- 識別子はコード内の対象を区別する名前
- 名前は意味を伝えるためのもの
- 短すぎる名前や曖昧な名前は避ける
- 命名は設計と可読性に直結する
- C# の慣習に沿った一貫性が重要
