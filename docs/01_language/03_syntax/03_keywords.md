# 03_keywords  ：予約語

## 概要

このファイルでは、C# の **予約語** を整理します。  
予約語とは、言語が特別な意味を持たせている単語のことです。

たとえば、`class`、`if`、`switch`、`namespace` などは、C# の文法そのものを表す重要な単語です。  
これらは自由に名前として使うのではなく、**言語機能を表す記号のようなもの** として扱います。

このファイルで扱う内容は次のとおりです。

- 予約語の意味
- 予約語の役割
- 識別子との違い
- `@` を使ったエスケープ識別子
- 実務での注意点
- よく使う予約語の見え方

---

## 定義

### 予約語とは何か

**予約語** は、C# が文法上の意味を持たせている単語です。  
これらは、変数名やメソッド名のような通常の識別子とは異なり、言語機能を表すために使われます。

例:

- `class`
- `namespace`
- `if`
- `else`
- `switch`
- `for`
- `while`
- `return`
- `public`
- `private`

### 識別子との違い

識別子は、開発者が自由に付ける名前です。  
一方、予約語は、C# があらかじめ意味を決めている単語です。

たとえば、`totalPrice` は識別子ですが、`class` は予約語です。  
この違いを理解すると、エラーの原因や文法の見え方が分かりやすくなります。

---

## 特徴

### 1. 文法の骨格を作る
予約語は、C# の構文を形作る中心的な単語です。  
`class` がないとクラス定義が書けず、`if` がないと条件分岐が書けません。

### 2. 名前としてそのまま使えない
予約語は通常の識別子としては使えません。  
つまり、変数名やクラス名としてそのまま使うと、コンパイルエラーになります。

### 3. `@` を付ければ使える場合がある
C# では、予約語と同じ文字列を識別子として使いたい場合、先頭に `@` を付けることでエスケープできます。  
ただし、これは最終手段に近く、通常は避けるほうがよいです。

### 4. C# の読みやすさに直結する
予約語を正しく理解していると、コードの構造が見えやすくなります。  
特に条件分岐、繰り返し、アクセス修飾子は頻出です。

---

## 代表例

### 1. クラス定義の予約語

```csharp
public class User
{
}
```

`public` と `class` は、C# の構造を表す代表的な予約語です。

---

### 2. 条件分岐の予約語

```csharp
if (score >= 60)
{
    Console.WriteLine("Pass");
}
else
{
    Console.WriteLine("Fail");
}
```

`if` と `else` は、分岐の流れを作るための予約語です。

---

### 3. 繰り返しの予約語

```csharp
for (int i = 0; i < 3; i++)
{
    Console.WriteLine(i);
}
```

`for` は、繰り返し処理を表します。

---

### 4. 名前空間の予約語

```csharp
namespace DemoApp.Services;
```

`namespace` は、型を整理するための予約語です。

---

### 5. アクセス修飾子

```csharp
private int _count;
public string Name { get; set; }
```

`private` や `public` は、外部からの見え方を制御する予約語です。

---

## 使いどころ

予約語の理解は、次の場面で特に役立ちます。

- C# のコードを読み始めるとき
- 文法エラーを見つけたいとき
- 予約語と識別子の違いを整理したいとき
- アクセス修飾子の意味を理解したいとき
- 条件分岐やループの構造を追いたいとき

実務では、予約語を「暗記する」のではなく、**構文の部品として自然に見えること** が大切です。

---

## 注意点

### 1. 予約語を安易に識別子として使わない
予約語をそのまま変数名などにすると、コードが読みにくくなったり、エラーになったりします。

### 2. `@` に頼りすぎない
C# では `@class` のように予約語を識別子として使えますが、見た目が不自然になりやすいです。  
特別な理由がないなら、別の名前を付けるほうがよいです。

### 3. 似た単語と混同しない
`readonly`、`record`、`required` のような単語は、予約語やコンテキストキーワードとして扱いが少し異なる場合があります。  
この章ではまず「文法に使う特別な単語」として押さえるのがよいです。

### 4. 見慣れない単語でも文法の役割を確認する
C# は進化する言語なので、新しいキーワードが増えることがあります。  
見慣れない単語を見たら、その役割を文法として確認する習慣が重要です。

## 実装例

### 例1: 予約語の基本的な使い方

```csharp
public class Sample
{
    public void Execute()
    {
        if (true)
        {
            return;
        }
    }
}
```

この例では、`public`、`class`、`void`、`if`、`return` がそれぞれ文法上の役割を持っています。

---

### 例2: 予約語を識別子として使う場合

```csharp
var @class = "Demo";
Console.WriteLine(@class);
```

`class` は予約語ですが、`@` を付けることで識別子として使えます。  
ただし、通常は別の名前にするほうが自然です。

---

### 例3: 名前の衝突を避ける

```csharp
string @namespace = "DemoApp";
```

このように書くことはできますが、実務では避けるのが一般的です。  
`appNamespace` のような別名のほうが読みやすいです。

## 設計指針

### 1. 予約語は役割で覚える
単語そのものを丸暗記するより、何を表しているかで整理します。

### 2. 識別子と混同しない
名前を付ける側と、文法として使う側を区別します。

### 3. `@` は例外扱いにする
予約語と同名の識別子は、基本的には避けます。  
どうしても必要なときだけ使います。

### 4. 予約語の理解を構文理解の入口にする
`if`、`for`、`class`、`namespace` などは、以降の章でも繰り返し出てきます。  
ここでしっかり押さえると、後の理解がかなり楽になります。

---

## 判断基準

予約語をどう扱うかは、次の基準で判断します。

### 1. 文法上の意味があるか
意味があるなら予約語として扱います。

### 2. 名前として使うべきか
通常は使わず、別名を選ぶほうがよいです。

### 3. `@` を付ける必要があるか
必要がなければ使いません。

### 4. 可読性が損なわれないか
予約語の扱いが不自然だと、コードの見通しが悪くなります。

---

## よくある誤解

### 1. 予約語は全部暗記しないといけない
最初からすべて覚える必要はありません。  
よく使うものから自然に慣れていけば十分です。

### 2. `@` を付ければ何でもよい
技術的には使えても、読みやすさは別問題です。  
使わないで済むならそのほうがよいです。

### 3. 予約語と識別子の違いは小さい
実際にはかなり重要です。  
この違いが曖昧だと、文法エラーの原因がつかみにくくなります。

### 4. 予約語は古い知識で十分
C# は進化しているため、新しいキーワードの意味も確認する必要があります。

---

## まとめ

予約語は、C# の文法を支える基本要素です。  
識別子と違い、開発者が自由に意味を決めるものではなく、言語が役割を与えている単語です。

このファイルで押さえるべき点は次のとおりです。

- 予約語は C# の構文を作る特別な単語
- 識別子とは役割が違う
- `@` で回避できる場合がある
- ただし、`@` は基本的に最終手段
- 予約語の理解は、以降の構文理解に直結する
